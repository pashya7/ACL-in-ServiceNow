01 — Core Concepts

ACL Building Blocks

📋

Table-Level ACL

Controls access to an entire table. No field specified. Determines if a user can see the table at all.

readwritecreatedelete
incident → read
🔒

Field-Level ACL

Controls access to a specific field on a table. Makes fields visible/hidden or editable/read-only.

readwrite
incident.priority → write
📄

Record-Level ACL

Controls access to specific records via conditions or scripts. "Users see only their own incidents."

conditionscript
incident → read + script
⚡

5 Operations

read, write, create, delete, execute. Each operation is evaluated independently by the ACL engine.

CRUDexecute
sys_security_operation
🔄

Evaluation Engine

Three checks per ACL: Role → Condition → Script. ALL must pass. Between ACLs: AND logic. Between roles: OR logic.

AND/ORhierarchy

Security Diagnostics page, Session Debug, ACL debug logging. Essential for "Access Denied" troubleshooting.

Specificity Order (Most → Least)
1. Field ACL on exact table
2. Field ACL on parent table
3. Table ACL on exact table
4. Table ACL on parent table
5. Wildcard ACL (*)
6. No match → default_mode
Logic Rules
Between multiple ACLs → AND (all must pass)
Between roles in ONE ACL → OR (any suffices)
Within ACL: Role + Condition + Script → AND
⚠️ Table Hierarchy: ACLs on parent tables (task) apply to child tables (incident). Both evaluate with AND logic — user needs to pass both.
04 — The Three Checks

ACL Permission Model

01

Role Check

Does user have at least ONE required role? Multiple roles = OR logic. Empty roles = everyone passes. Admin passes all (unless admin_overrides = false).

02

Condition Check

Does the record meet condition builder criteria? No condition = auto-pass. Evaluated against the current record using standard condition builder.

03

Script Check

Does the script set answer = true? No script = auto-pass. Has access to current record and all gs.* APIs. previous is NOT available.

🔑 Key Rule: ALL three checks must pass for ONE ACL. ALL matching ACLs must pass for access. Admin bypasses unless admin_overrides = false.

09 — Pro Tips

ACL Tips

01

Conditions Over Scripts

Use condition builder when possible. Conditions are compiled and faster. Scripts are interpreted on every single access.

02

No GlideRecord in ACL Scripts

Avoid querying tables inside ACL scripts. Use gs.getUser().isMemberOf() instead of querying sys_user_grmember.

03

Always Impersonate to Test

Admin bypasses ALL ACLs. Testing as admin tests nothing. Always impersonate a non-admin user.

04

Security Diagnostics Is Your Friend

Navigate to sys_security_acl_debug.do — shows exactly which ACL passes or fails and why.

05

Check Parent Table ACLs First

ACLs on task table apply to incident, change_request, etc. Both evaluate with AND logic. Plan hierarchy carefully.

06

Don't Forget Execute ACLs

Client-callable Script Includes without execute ACLs are a security hole. Any authenticated user can call your GlideAjax.

07

admin_overrides = false

For truly sensitive data (PII, salary, HIPAA), set this to false. Even admin must satisfy conditions/scripts.

08

default_mode = deny

Set glide.sm.default_mode = deny in production. Tables without ACLs should be blocked, not open.

09

Test via REST API Too

ACLs must protect both UI and API access. Use Postman to verify your ACLs work for API calls.

10

Right-Click Field Shortcut

Right-click any field on a form → Configure → Security Policy. Shows ACLs for that specific field instantly.

10 — Best Practices

Do's & Don'ts

✅ Best Practices

✅

Always specify required roles — never empty

✅

Test by impersonating non-admin users

✅

Use condition builder over scripts when possible

✅

Keep ACL scripts under 10 lines

✅

Document every custom ACL with description

✅

Set default_mode = deny in production

✅

Check parent table ACLs before creating new ones

Test Yourself

Score
Q01 Easy
Which table stores ACL rules in ServiceNow?
A sys_security_rule
B sys_security_acl
C sys_acl
D sys_access_control
✅ B — sys_security_acl stores all ACL rule definitions. Supporting tables include sys_security_acl_role for role links.
Q02 Easy
Name the 5 ACL operations.
A get, set, add, remove, run
B select, insert, update, delete, call
C read, write, create, delete, execute
D view, edit, new, remove, process
✅ C — read, write, create, delete, execute. Write ≠ create. Execute covers processors, GlideAjax, and REST endpoints.
Q03 Medium
When multiple ACLs match the same request, what logic is used BETWEEN them?
A AND — all must pass
B OR — any one passes
C First match wins
D Most specific wins
✅ A — AND logic. ALL matching ACLs must pass. Within a single ACL, multiple roles use OR logic. Between ACLs = AND.
Q04 Medium
What happens when NO ACL matches and glide.sm.default_mode = 'deny'?
A Access granted
B Access denied
C Error thrown
D Admin is asked
✅ B — Access denied. With deny mode, unprotected tables are blocked by default. This is the recommended production setting.
Q05 Medium
True or False: The admin role bypasses ALL ACLs in ALL cases.
A True
B False
✅ False — Admin bypasses ACLs where admin_overrides = true (default). Set admin_overrides = false to block even admins.
Q06 Hard
What is the evaluation order within a SINGLE ACL rule?
A Script → Condition → Role
B Condition → Role → Script
C Role → Condition → Script
D All checked simultaneously
✅ C — Role → Condition → Script, in that order. ALL must pass (AND logic). If role fails, condition and script aren't evaluated.
Q07 Hard
You create a field-level WRITE ACL on incident.priority requiring 'priority_manager'. A user with only 'itil' opens an incident. What happens?
A Priority field is hidden
B Entire form is blocked
C Priority field is read-only
D Error message shown
✅ C — Read-only. The field is visible (read ACL passes with itil) but not editable (write ACL requires priority_manager).

🏢 GlobalTech Corp — IT Asset Request Security

Build a complete ACL security model for a custom application — by Prasad Shelar

"You are a ServiceNow developer at GlobalTech Corp. The security team needs ACLs for your custom 'IT Asset Request' table. 'Lock it down properly — we can't fail another audit.'"

🥉 Bronze
  • Create custom table: u_asset_request
  • Add fields: requester, asset_type, status, cost, notes
  • Create 3 roles: x_asset.viewer, .requester, .admin

  • Test with impersonation
🥈 Silver
  • Field ACL: cost → read (only x_asset.admin)
  • Field ACL: notes → write (only x_asset.admin)
  • Record ACL: requesters see only own requests
  • Verify with Security Diagnostics page
🥇 Gold
  • Before Query BR for row-level security
  • Business hours ACL (write only Mon-Fri 8-6)
  • admin_overrides=false ACL (prevent delete)
  • Complete test matrix for all roles
⭐ Bonus
  • Execute ACL on GlideAjax Script Include
  • Test all ACLs via REST API (Postman)
